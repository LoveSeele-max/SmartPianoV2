# **🎹 智能钢琴练习助手 \- V2 专属优化方案总结**

基于“仅适配个人雅马哈 PSR-E383”的原则，V2 版本的核心优化方向已剔除不必要的硬件兼容需求，聚焦于**性能、便捷度与代码架构**。

## **一、 性能优化：渲染引擎重构（优先级：⭐⭐⭐⭐⭐）**

**当前痛点**：卷帘窗基于 DOM 渲染（每个音符一个 \<div\>）。如果导入一首包含几千个音符的 MIDI 练习曲，会瞬间生成大量 DOM 节点，配合进度条滚动会导致严重的掉帧和卡顿。

**优化方案**：

* **迁移至 Canvas API**：废弃现有的 div 堆叠方式，使用 \<canvas\> 绘制卷帘窗。  
* **按需绘制**：配合 requestAnimationFrame，每帧只计算并绘制当前时间轴视口内（屏幕可见范围）的音符。无论曲子有多长、多复杂，渲染性能开销都始终保持在极低水平，实现丝滑滚动。

## **二、 便捷度提升：专属本地化功能（优先级：⭐⭐⭐⭐）**

**当前痛点**：每次练琴都需要手动点击上传曲谱，且 Soundfont 真实音色库依赖网络动态加载，弱网环境下第一次按键会没声音或有延迟。

**优化方案**：

* **本地曲谱库 (IndexedDB / localStorage)**：做一张简单的“播放列表” UI。用户上传过的曲子自动保存在浏览器本地数据库中。下次打开网页直接从列表点击播放，无需再次寻找本地文件上传。  
* **音色资源离线缓存**：利用 **Service Worker** 或 **Cache Storage API**，将 acoustic\_grand\_piano 音色文件永久缓存在浏览器中。实现“二次秒开”甚至在断网环境下也能正常使用。

## **三、 交互细节修复：杜绝按键卡死（优先级：⭐⭐⭐）**

**当前痛点**：鼠标在虚拟键盘上滑动时，基于 mousedown / mouseup / mouseleave 的事件组合容易丢失状态，导致某些琴键一直处于红色高亮（按下）状态无法弹起。

**优化方案**：

* **引入 Pointer Events**：将原本的鼠标事件替换为 pointerdown、pointerup 和 pointercancel。  
* **全局释放保障**：在 window 上绑定全局的 pointerup 事件。只要用户松开鼠标，无论指针在哪里，都强制清理所有“卡住”的琴键状态。

## **四、 代码架构整改：解耦与模块化（优先级：⭐⭐⭐）**

**当前痛点**：目前近 600 行的 HTML、CSS 和 JavaScript 业务逻辑全部耦合在单个 SmartPiano.html 文件中。后续如果添加复杂曲谱解析或 Canvas 逻辑，代码将极难阅读和维护。

**优化方案**：进行合理的文件拆分（通过 ES6 Modules 引入）：

1. index.html：仅保留 Tailwind UI 骨架和基础 DOM。  
2. audioEngine.js：专职管理 Web Audio API、Soundfont 加载、压限器 (DynamicsCompressor) 及节点生命周期管理。  
3. midiController.js：专职管理 Web MIDI API 连接、雅马哈专属的 254/248 信号过滤、以及 MIDI 信号解析。  
4. parser.js：存放 MusicXML、MIDI 和 JSON 的纯文本/二进制解析逻辑。  
5. app.js：核心控制中枢，负责串联 UI、播放状态机（练习模式/自动播放）、节拍器和进度条逻辑。

**💡 开发者备忘：** \> 针对雅马哈 PSR-E383 的特殊过滤规则 (cmd \=== 254 和 cmd \=== 248\) 已经非常完善，务必在模块化重构时保留此核心逻辑，防止琴端的时钟信号导致主线程阻塞。
